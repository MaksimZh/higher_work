# Interface Segregation Principle

## 1. Добавление узла дерева в ECS

### Было
Функция имеет полный доступ к реестру и может удалить сущности, нарушив работу
других модулей:
```C++
void addChild(ecs::Registry& registry, Entity parent, Entity child) {
  assert(!registry.has<Parent>(child));
  registry.insert(child, Parent{parent});
  if (!registry.has<FirstChild>(parent)) {
    registry.insert(parent, FirstChild{child});
    registry.insert(parent, LastChild{child});
    return;
  }
  auto& last = registry.get<LastChild>(parent);
  registry.insert(last.entity, NextSibling{child});
  registry.insert(child, PrevSibling{last.entity});
  last.entity = child;
}
```

### Стало
Функция требует только те методы, которые действительно использует.
Да, можно использовать и другие, и при подходящем типе это скомпилируется,
но это уже будет какой-то шаблонный downcast.
```C++
template<typename R, typename E>
concept ComponentRegistry = requires(R& r, const R& cr, E entity, E other) {
  { cr.template has<int>(entity) } noexcept -> std::same_as<bool>;
  { r.insert(entity, int{}) } noexcept -> std::same_as<void>;
  { r.template get<int>(entity) } noexcept -> std::same_as<int&>;
};

template <typename Registry, typename Entity>
  requires ComponentRegistry<Registry, Entity>
void addChild(Registry& registry, Entity parent, Entity child) {
  assert(!registry.template has<Parent>(child));
  registry.insert(child, Parent{parent});
  if (!registry.template has<FirstChild>(parent)) {
    registry.insert(parent, FirstChild{child});
    registry.insert(parent, LastChild{child});
    return;
  }
  auto& last = registry.template get<LastChild>(parent);
  registry.insert(last.entity, NextSibling{child});
  registry.insert(child, PrevSibling{last.entity});
  last.entity = child;
}
```


## 2. Удаление поддерева

### Было
По сути, то же, что и в предыдущем случае, только появляется `remove`.
```C++
void detachChild(Registry& registry, Entity child) {
  if (!registry.has<Parent>(child)) {
    return;
  }
  Entity parent = registry.get<Parent>(child).entity;
  registry.remove<Parent>(child);

  ...
}
```

### Стало
Просто добавил `remove`, хоть он и не нужен, например, в `addChild`,
но дробить концепт дальше имеет смысл только один раз - на отдельные функции,
из которых потом будут собираться требования для каждого шаблона.
Интуиция говорит, что такая крайность - это перебор,
а любой промежуточный вариант лишь увеличит сложность без явной пользы.

Удаление компонента, на самом деле, не такая уж опасная операция -
не хуже перезаписи.
Модуль для управления деревом имеет доступ только к своим типам компонентов,
и если что-то и испортит, то только самому себе.
Что действительно опасно, так это удаление сущности,
которое может уничтожить компоненты других модулей.
```C++
template<typename R, typename E>
concept ComponentRegistry = requires(R& r, const R& cr, E entity, E other) {
  { cr.template has<int>(entity) } noexcept -> std::same_as<bool>;
  { r.insert(entity, int{}) } noexcept -> std::same_as<void>;
  { r.template get<int>(entity) } noexcept -> std::same_as<int&>;
  { r.template remove<int>(entity) } noexcept -> std::same_as<void>;
};

template <typename Registry, typename Entity>
  requires ComponentRegistry<Registry, Entity>
void detachChild(Registry& registry, Entity child) {
  if (!registry.template has<Parent>(child)) {
    return;
  }
  Entity parent = registry.template get<Parent>(child).entity;
  registry.template remove<Parent>(child);

  ...
}
```


## 3. Scorer

### Было
```C++
class Scorer {
 public:
  ...

  // -- Accessors used by the page feature extractor ---------------------------

  // Returns a callback to find if a page word is in the model.
  base::RepeatingCallback<bool(uint32_t)> find_page_word_callback() const;

  // Returns a callback to find if a page term is in the model.
  base::RepeatingCallback<bool(const std::string&)> find_page_term_callback()
      const;

  ...
};
```

Вообще-то, это не относится к "скорингу".
Extractor обращается к этой части Scorer, чтобы получить данные,
которые затем передаются в Scorer.

Когда удалили экстрактор, эти методы тоже убрали,
потому что они вовсе не были частью интерфейса Scorer.


### Стало
```C++
class PhishingModel {
 public:
  virtual bool HasPageWord(uint32_t hash) const = 0;
  virtual bool HasPageTerm(const std::string& term) const = 0;
  ...
};
```
А в нашем форке Extractor нужен, а Scorer у нас свой.
Поэтому для экстрактора нужен отдельный интерфейс
со вспомогательными функциями.


## 4. Host

## Было
```C++
class Host : ... {
  ...
 private:
  ...
  void StartDetection(..., callback);
  void FinishDetection(..., callback);
  ...
};
```
Это приватный интерфейс.
Они используется внутри самого класса, и проблема здесь в том,
что эти функции имеют доступ ко всем полям, хотя нужна им всего пара-тройка.

Это нужно не просто вынести в отдельный интерфейс,
тут и реализацию нужно вынести в отдельный класс.


## Стало
```C++
class Detector {
  ...
  void StartDetection(..., callback);
 private:
  void FinishDetection(..., callback);
  ...
};
```
Host будет вызывать только один метод Detector,
а "лишние" методы будут приватными.

Detection можно реализовать как чистый dataflow,
так почему всё же класс, а не функция?
Потому что это реализовано как серия операций, которые запускаются асинхронно,
и в которых длинные задачи вроде парсинга текста разбиваются на куски
по времени, и встают на паузу чтобы браузер не тормозил.
Это такая квантовая многозадачность своими руками.

Но самое главное - это всё должно поддерживать отмену выполнения на любом этапе.
В Chromium это традиционно делается через разделяемое состояние
со слабыми ссылками.
Когда операция стартует с протухшей ссылкой, она отменяется
и отменяет все следующие звенья цепочки.

Detector - это и есть то самое мастер-состояние,
которое владеет всеми разделяемыми данными
(ну не копировать же полный текст страницы)
и отвечает на вопрос "продолжаем? да/нет?".


## 5. WebContents

## Было
```C++
void Detector::StartDetection(..., callback) {
  ...
  content::RenderFrameHost* rfh = web_contents()->GetPrimaryMainFrame();
  current_url_ = rfh->GetLastCommittedURL();
  ...
  ExtractDomFeatures(rfh,  // фактически, берём из web_contents
                    ...);
  ...
  LaunchSomeNDAMethod(
      rfh->GetLastCommittedURL(),  // фактически, берём из web_contents
      web_contents()->GetTitle(),
      page_text,  // это тоже взяли из web_contents
      ...);
  ...
}
```
Так то `WebContents` только в заголовочном файле -
сотни строк кода не считая комментариев-документации.
И после рефакторинга хоста в предыдущем пункте передавать его в
детектор целиком, давая полный доступ практически ко всей вкладке браузера?

Вытащить все данные заранее и передать 5 параметров?
А 3 из них получаются асинхронно, и запросить их нужно в определённый момент.
Именно Detector должен решать когда забирать данные.

Нам нужен интерфейс, который открывает часть web_contents()
и частично использует уже загруженные данные.


## Стало
```C++
void Detector::StartDetection(PageDataExtractor* page_data_extractor,
                              Scorer* scorer,
                              DetectorDoneCallback done) {
  page_data_extractor->GetDOMFeatures(...);
  ...
  Chain(
    Bind(&PageDataExtractor::GetText, page_data_extractor),
    Bind(&LaunchSomeNDAMethod,
         page_data_extractor->GetUrl(),
         page_data_extractor->GetTitle())
    ...).Run();
  ...
}
```
`PageDataExtractor`, `Scorer`, `DoneCallback` - это порты для `Detector`.
`Host` должен будет предоставить адаптеры, например, для передачи данных из
`WebContents` через `PageDataExtractor`.
