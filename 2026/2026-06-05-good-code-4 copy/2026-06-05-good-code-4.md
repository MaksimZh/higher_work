# 4 важных свойства хорошего кода
1. Хороший код можно (и нужно) понимать модульно.
2. Хороший код позволяет быстро восстановить замысел программиста.
3. Хороший код выражает замысел разработчика в одном месте.
4. Хороший код надёжен/робастен
  (несильные изменения требований подразумевают несильное изменение кода).

## 1. Старая версия пет-проекта

### 1.1 Хороший код можно (и нужно) понимать модульно.
50%

Есть разделение на пакеты (ECS, GUI, ...) внутри код сгруппирован
в подпакеты и модули.

В ECS пришлось вносить изменения
для поддержки работы GUI на отдельных платформах.
Модули, которые должны быть одним целым - окна и OpenGL - были оформлены как
отдельные системы и обменивались данными через ECS (низкоуровневые интерфейсы
в открытом доступе).

### 1.2. Хороший код позволяет быстро восстановить замысел программиста.
70%

В ECS хорошее разделение по уровням абстракции,
технические детали скрыты и не мешают восприятия общего замысла,
это делает код компактным и легко понимаемым.

В GUI попытка сделать то же самое, но абстракция поверх оконной системы ОС
слабая - она "протекает", заставляет думать об окнах и их отношениях
с контекстом OpenGL.

### 1.3. Хороший код выражает замысел разработчика в одном месте.
80%

В GUI постарался локализовать платформозависимый код,
и код, зависящий от графического API - тоже.
Внутри модуля для Windows каждый сценарий обработки сообщений
тоже вынесен в отдельный модуль.

При этом, правда, есть сценарии, которые потребовали вносить изменения
аж в ECS - другой пакет, но в новой версии я это починил.

### 1.4. Хороший код надёжен/робастен
60%

Попытка засунуть в ECS ресурсы, требующие контроля времени жизни,
привела к тому, что программа падала при завершении.
Изменения в ECS пришлось вносить для поддержки
контролируемого освобождения ресурсов.
Это, конечно, не отменяет необходимости аккуратно работать с этой системой
в клиентском коде.
Ну и зачем?

В новой версии весь менеджмент отдан клиентскому коду,
а ECS - просто хранилище данных, а не уникальных сущностей.
Поддержку move-semantics я всё же оставил,
но это для обычных данных, которые просто не хотим лишний раз копировать,
а не для каких-то уникальных RAII объектов.


## 2. Старый модуль в рабочем проекте

### 2.1 Хороший код можно (и нужно) понимать модульно.
10%

Регулярно запутываешься во вкладках IDE, потому что, чтобы понять как работает
отдельная функция, нужно отследить цепочку вызовов через несколько модулей.
Особенно весело, когда натыкаешься на границу в виде абстрактного интерфейса,
который только мешает, потому что контракт отсутствует (или нарушается),
а найти реализацию, которая подставляется в данном месте - это ещё цепочка
из 5 вызовов в трёх разных модулях.

### 2.2. Хороший код позволяет быстро восстановить замысел программиста.
50%

После некоторого обучения на данной кодовой базе, начинаешь видеть
повторяющиеся шаблоны, которые используются для решения типовых задач.
При этом есть более наглядные и понятные новичкам варианты, но для этого же
надо было добавить в код какие-то функции, или типы данных, а это тяжело.

Особенно трудно с новыми типами: например стандарты кодирования в проекте
для каждого типа сложнее `point(x,y)` требуют явно определит конструктор и
деструктор вне заголовочного файла.


### 2.3. Хороший код выражает замысел разработчика в одном месте.
50%

Если нормировать на масштаб проекта, то это выполняется.
Только "одно место" может оказаться пакетом размером с небольшой стартап,
состоящим из нескольких десятков модулей.

### 2.4. Хороший код надёжен/робастен
40%

Тут как в анекдоте: переставили фонарь, и Пьер Безухов погиб.
Из свежего.
Сервис должен вызвать колбек, который передан клиентом,
это могло бы выглядеть как вызов чистой функции: запрос -> результат/ошибка,
но нет.
При некоторых ошибках возвращается ошибка, а при некоторых других -
колбек сохраняется внутри сервиса, а при следующем запросе,
при попытке замены новым колбеком, старый вызывается с ошибкой "отмена".
Потом некоторые тесты, которые отправляют ровно один запрос, зависают.


## 3. Новый модуль в рабочем проекте

### 3.1. Хороший код можно (и нужно) понимать модульно.
70%

Как ни трудно вставлять новые модули в систему, где есть аж две независимых
подсистемы для отслеживания зависимостей, но оно того стоит.
От цепочек колбеков пока не удалось полностью избавиться:
нужные абстракции внедряются постепенно и не без сопротивления.

### 3.2. Хороший код позволяет быстро восстановить замысел программиста.
70%

Добавлены новые типы и функции, которые позволяют писать код понятными словами
на более высоком уровне абстракции.

### 3.3. Хороший код выражает замысел разработчика в одном месте.
70%

Каждому замыслу - свой модуль.
Если нужно вызвать цепочку колбеков, то она строится в одном месте -
в точке первого вызова, а не скрывается внутри функций.

### 3.4. Хороший код надёжен/робастен
70%

Стараюсь вообще не использовать явные изменяемые состояния.
Никаких зависимостей запросов друг от друга,
исключение делается только для кеширования данных.
Это снимает кучу проблем с асинхронным кодом.
