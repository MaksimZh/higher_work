## 1. ECS и окна
Делаю GUI с поддержкой нескольких системных окон (чего нет почти нигде),
да такой, чтобы использовал ECS (свой, или интегрировался в существующий).

Создали мы несколько окон, теперь их надо настроить,
а этим занимаются другие системы.
Они, конечно, могут перечислить все оконные сущности,
работая с реестром напрямую:
```
view : Registry -> ComponentType -> List[Entity]
```
Реестр - тоже абстракция, над памятью.
Но прямой доступ к нему не намного лучше прямого доступа к памяти.
Можно случайно удалить чужую сущность, или переписать любой компонент,
если доступен его тип.
Для поиска новых окон пришлось бы дать доступ к `NewWindowMarker`, например.

Поднимаемся на новый уровень абстракции:
```
new_windows : Registry -> List[Entity]
```
Вместо "найди сущности с компонентом `NewWindowMarker`",
говорим "дай все сущности - новые окна".
Так можно не отдавать свои компоненты в другую систему.

В идеале, можно обложить системы абстракциями и вообще не пускать их к реестру.
Получится, что каждая система будет жить в своём мире с абстракциями-портами,
а снаружи будут адаптеры, которые "поднимают" реестр на нужный уровень.

Не съест ли это все плюсы производительности ECS?
Нет, если делать через шаблоны, а не виртуальные интерфейсы.
Тогда компилятор сам почистит лишние слои.


## 2. TfLite
Работа с TfLite в C++ - это всегда весело:
```C++
instance->model = tflite::FlatBufferModel::VerifyAndBuildFromBuffer(
    model_data.data(), model_data.size());
instance->resolver = std::make_unique<tflite::ops::builtin::BuiltinOpResolver>();
tflite::InterpreterBuilder(*instance->model, *instance->resolver)(
    &instance->interpreter);
instance->interpreter->AllocateTensors();
```
А с правильной обработкой ошибок - ещё веселее.

Поднимаемся на верхний уровень:
```C++
auto model = TfLiteModelWrapper<tuple<int, int>, tuple<float>>::FromFlatBuffer(buffer);
// model - expected<T, E>, но мы считаем, что получилось:
auto result = model.value()->Invoke(mask, tokens);
```
Все эти ресолверы, интерпретаторы, проверка типов тензоров - они больше
не в одной комнате с нами.
Здесь мы работаем с моделью в терминах "создать", "вычислить", "ошибка".


## 3. Не пара
Это почти псевдокод, потому что для читаемости
я избавился от синтаксической "соли", характерной для C++ и Chromium.
```C++
template <typename T1, typename T2>
Callback<void(T1, T2)> UnwrapArgs(Callback<void(pair<T1, T2>)> callback) {
  return Bind(
      [](Callback<void(pair<T1, T2>)> cb, T1 first, T2 second) {
        cb(first, second);
      },
      callback);
}
```
Эта функция снимает обёртку `pair` с аргументов колбека.
Можно сказать, что она осуществляет преобразование в эквивалентную функцию
другого типа:
```
UnwrapCallback: ( (T1, T2) -> () ) -> ( T1 -> T2 -> () )
```
Это и есть более высокий уровень абстракции,
который легко читается в клиентском коде.

На низком уровне - стандартная хромовская кухня с Bind, лямбдой и кучей `move`,
которые я выпилил для наглядности.
Когда такое встречается в клиентском коде, восприятие логики сразу прерывается,
и начинаются размышления: "а что это вообще".
Через несколько месяцев мозг может читать это между строк, но зачем?


## 4. Timeout
Снова приукрашу, в реальности оно страшнее:
```C++
template <typename... Args>
Callback<void(Args...)> MakeTimeoutCallback(
    Callback<void(Args...)> callback,
    TimeDelta timeout,
    Args... timeout_args) {
  auto trigger = MakeTriggerCallback(callback);
  PostDelayedTask(Bind(trigger, timeout_args...),
                  timeout);
  return trigger;
}
```
Триггер - уже абстракция.
Его можно копировать, и вызов любой копии передаёт управление дальше,
а вызовы остальных копий игнорируются.
Внутри ещё два уровня абстракции: самодельный счётчик вызовов и
хромовский контейнер с подсчётом ссылок.

А теперь добавим ещё абстракцию.  
Верхний уровень: создать колбек, который будет вызван либо вручную,
либо автоматически через отведённое время с параметрами по умолчанию.  
Нижний уровень: создать триггер, который вызовет либо тот, для кого он создан,
либо отложенная задача, в которую мы загрузили параметры по умолчанию.
